# ADR-0075 ‚Äî Shift+Tab mode cycle (normal ‚Üí auto-accept-edits ‚Üí plan)

**Status:** Accepted (refines ADR-0051, ADR-0055)

**Context.** zrb already had the two interaction modes Claude Code's `Shift+Tab`
toggles between, but as two **independent** controls: plan mode (`/plan`,
`Ctrl+P`, the `EnterPlanMode` tool ‚Äî a read-only `PLAN_MODE_POLICY`, ADR-0051)
and yolo (`/yolo`, `Ctrl+Y` ‚Äî auto-approval, `bool | frozenset[str]`, ADR-0055).
Users coming from Claude Code reached for a single key that cycles through an
ordered set of modes with a persistent badge, and found neither the gesture nor
the indicator. Two gaps stood out: there was no **one** key to step through the
modes, and there was no first-class **"auto-accept edits"** middle mode (yolo was
all-or-an-arbitrary-set, never surfaced as a named state) ‚Äî even though selective
yolo can already express exactly that.

**Decision.** Add a single `Shift+Tab` (`s-tab`) key binding that cycles an
ordered list of three modes, reusing the existing enforcement machinery ‚Äî no new
policy, no new permission primitive:

| Cycle step | plan store | yolo store |
| --- | --- | --- |
| `normal` | BUILD | `False` |
| `accept_edits` | BUILD | `frozenset({"Write", "Edit"})` |
| `plan` | PLAN | `False` |

- **State is derived, not stored.** `ModelCommandsMixin.current_cycle_mode()`
  reads the two live stores (`_plan_mode_active` + `yolo`) and returns the mode
  name; `cycle_mode()` advances to the next member and `_apply_cycle_mode()`
  writes **both** stores so the cycle keeps them mutually exclusive (plan and
  auto-approve never stack). No third "current mode" cursor exists to drift out
  of sync with the stores that actually gate tools.
- **`accept_edits` is selective yolo** over the LLM-visible edit-tool names
  (`write_file.__name__ == "Write"`, `replace_in_file.__name__ == "Edit"`),
  matched by the existing `tool_name in yolo_value` check in
  `chat/task.py::check_yolo`. File writes auto-approve; every other tool (shell,
  delegation, fetch) still prompts.
- **Off-cycle yolo resets predictably.** Full yolo (`True`) or a custom set
  (`/yolo Read,Bash`) is reported as `yolo`/`custom` and is *not* a cycle member;
  `Shift+Tab` from there re-enters at `normal`, so the gesture is always defined.
- **Persistent badge.** The default UI status bar renders the current mode name
  and a `shift+tab to cycle` hint (`output_mixin.get_status_bar_text`), mirroring
  Claude Code's mode indicator near the prompt. The info bar's existing plan/yolo
  lines are unchanged (they remain truthful for off-cycle states).
- **Freed from focus traversal.** `Shift+Tab` was already bound ‚Äî the input and
  output controls bound `Tab`‚Üífocus-next / `Shift+Tab`‚Üífocus-previous
  (`app/layout.py`, `app/keybinding.py`). Control/widget bindings sit at the
  **highest** precedence layer (prompt-toolkit reverses the merge order so the
  focused control wins, `application.py::_create_key_bindings`), so an app-level
  `Shift+Tab` was silently shadowed. Rather than fight precedence with `eager`,
  the two control-level focus bindings were **removed**:
  `Tab` and `Shift+Tab` both cycle modes at the app level (gated
  `no_active_choice & ~has_completions`), since on Termux both keys
  produce the same byte (`0x09`) and cannot be distinguished; `Ctrl+K`
  becomes the sole focus-toggle binding, gated unconditionally.
  `F6` was the original focus-toggle key but was replaced by `Ctrl+K`
  because Termux maps F-keys to system actions ‚Äî `Ctrl+K` avoids that
  conflict without introducing conflicts with Emacs-mode or Vi-mode
  text-editing bindings (it overrides only Emacs-mode kill-line, which
  is less critical than focus traversal). The filter on Tab/Shift+Tab
  ensures a completion menu keeps Tab for next-completion and
  Shift+Tab for previous-completion, and an `AskUserQuestion` choice
  widget keeps its own Tab/back-tab navigation.
- **`Ctrl+P` retired; slash commands kept.** The cycle reaches plan mode, so the
  redundant `Ctrl+P` toggle was dropped. `/plan` and `/yolo` remain ‚Äî they are the
  only way to reach these modes in the web UI / MultiUI (no key chords there) and
  are scriptable. `Ctrl+Y`, `EnterPlanMode`/`ExitPlanMode` are unchanged. `/help`
  was updated to match.

**Consequences.** One muscle-memory keystroke matches Claude Code, and
"auto-accept edits" becomes a named, displayed state instead of folklore. Cost:
the cycle imposes a linear order on two orthogonal toggles, so it cannot express
every combination (e.g. plan + a custom yolo set) ‚Äî those remain reachable only
via the explicit controls, by design. The binding is CLI-only (the default
prompt-toolkit `Application`); web/MultiUI users still drive modes via `/plan`
and `/yolo Write,Edit`. A `/mode` slash-command for cross-surface parity is a
noted follow-up.

**Alternatives rejected.** Win the `Shift+Tab` precedence fight with `eager=True`
on the app-level binding ‚Äî rejected; it works, but leaves two contradictory
`Shift+Tab` handlers registered (focus + cycle) and relies on a subtle priority
rule. Removing the now-unwanted control-level focus bindings is clearer and
leaves one handler per key. Keeping `Ctrl+P` as a direct one-press plan toggle
alongside the cycle ‚Äî rejected as redundant surface once the cycle reaches plan;
`/plan` covers the non-CLI and scriptable cases. A new `AgentMode` enum value for
auto-accept-edits ‚Äî unnecessary; selective yolo already encodes it, and a third
enum would split one concept across two stores. A stored "current mode" cursor ‚Äî
rejected as a third source of truth that can disagree with the stores tools
actually consult.
