# ADR-0083 — Model-adaptive prompt profiles: per-section variants, declared (not guessed) per model

**Status:** Accepted (extends ADR-0061 prompt section composition)

**Context.** zrb's `PromptManager` composes the system prompt from ordered,
MECE sections (ADR-0061): `include_sections` decides *which* sections appear and
in what order, each section resolving to a markdown file or a registered
provider. The phrasing of each section is fixed — there is one `persona.md`, one
`mandate.md`, etc., written in a dense, principle-led register.

zrb is explicitly meant to run on both weak and strong models (the prompt is
also Claude-flavored on purpose, for skill/tooling compatibility with Claude
Code). A study of opencode's prompts (`session/prompt/*.txt`, selected by
`session/system.ts::provider()`) surfaced two findings that the current
single-register composition cannot express:

1. **opencode selects a whole hand-tuned monolith per model family**
   (`anthropic`, `gpt`, `gemini`, `kimi`, `codex`, `beast`, `default`) by
   substring-matching the model id. It does *not* compose. The cost is heavy
   duplication — the same safety rules ("never commit unless asked", "never
   revert user changes") are copy-pasted across nearly every file — which is
   exactly the MECE violation zrb's composition avoids.
2. **Weak ≠ fewer instructions.** The prompt for the *weaker* family
   (`beast.txt`, GPT-4/o1/o3) is the **longest and most repetitive** — it
   hammers persistence ("keep going", "ACTUALLY make the tool call") and is full
   of few-shot `<example>` blocks. The strong-model prompt (`anthropic.txt`) is
   **shorter** and leans on the model's judgment. The right axis is not
   *amount* but *register*: weak models need explicit, repeated, exemplified
   imperatives; strong models need terse, principled, high-trust prose. zrb's
   current prompt is ~6,900 tokens of dense principle — well-suited to a strong
   model, lossy on a weak one (which silently drops the middle of a long
   nuanced rulebook).

So the gap is: zrb can vary *what topics* appear, but not *how each topic is
phrased for the model in use*, and it has no default that adapts to model
strength. The aim is a system that "mostly works by default" — the user sets
nothing and gets a sensible prompt for their model.

**Decision (proposed).** Add a **second axis** to prompt assembly and an
**auto-mapping default**, keeping composition (zrb's strength) and borrowing
selection (opencode's strength) only where behavior genuinely doesn't decompose.

1. **Register variants (new axis).** Section content resolves to a
   register-specific variant when one exists, falling back to the base file:
   `get_prompt("persona", profile="explicit")` tries `persona.explicit.md`,
   then `persona.md`. Registers are capped at **two or three** (`terse` /
   `explicit`, optionally `balanced`) — deliberately **not per-vendor**, to
   avoid re-importing opencode's monolith duplication. Only register-sensitive
   sections need a variant; everything else uses the single shared file.

2. **`examples` as a real, toggleable component.** Few-shot `<example>` blocks
   are the single highest-leverage weak-model addition and are near-absent from
   zrb today. Ship them as a component that is on for `explicit`, off for
   `terse`.

3. **Register selection: declared, not guessed.** `ZRB_LLM_PROFILE`
   (`terse`/`explicit`/`auto`, default `auto`) sets the register. The register
   additionally composes the `examples` section under `explicit`. Crucially,
   `auto` does **not** infer strength from the model id. The id cannot reveal
   capability: family names (`deepseek`, `qwen`, `llama`, …) span tiny instruct
   models through frontier models, so any substring/family match *guesses* — as
   an early draft did, misclassifying DeepSeek V4 Pro as weak. This contradicts
   how zrb already handles model facts: `capabilities.py` curates a per-model
   table with a user-override hook and conservative defaults for unknowns,
   precisely because "there's no standard API to discover capabilities at
   runtime across providers." So `auto` resolves to the `terse` base unless a
   per-model profile has been **declared** via `register_model_profile(pattern,
   profile)` — a small user-extensible registry mirroring `model_capabilities`
   (the pattern matches the full model id as configured, nothing stripped).
   Weaker models are opted in explicitly (the knob, or a declaration), never
   auto-detected. This keeps "mostly works by default" honest: the default is the
   no-regression terse prompt, and adapting to a weak model is one deliberate
   setting, not a fragile heuristic.

4. **A small selection escape hatch for cross-cutting voice.** A few behaviors
   do not decompose into orthogonal toggles because they permeate every line
   (opencode's gpt `commentary`/`final` response channels, beast's recursive
   web-research mandate, a vendor formatting dialect). These stay expressible as
   a whole alternate section/preset rather than being forced into the register
   matrix.

Decomposition guide (from the opencode study), by how each behavior is handled:

- **Shared, single file** — convention-following, git/commit safety, environment
  block, AGENTS.md handling, search-via-subagent, tool parallelism.
- **Register-sensitive, variant file** — autonomy/persistence, conciseness,
  code-comment policy, todo cadence, verify/test intensity, and `examples`.
- **Cross-cutting, selection** — response channels, web-research mandate,
  vendor formatting dialect.

**Consequences.** zrb gets register-tuned prompts with one knob, while
preserving MECE composition and avoiding opencode's duplication (shared rules
stay shared; only register-sensitive sections fork, into ≤3 variants). The
default (`auto` with no declarations → `terse`) is a strict no-op for existing
setups. New surface: a register parameter through `get_prompt`/`PromptManager`,
the `model_profile_registry` + `register_model_profile` hook, an `examples`
component, and the first `terse`/`explicit` variant pair (`persona`). Risks:
variant drift (a rule updated in `terse` but not `explicit`) — mitigated by
keeping the variant set small and the shared/variant split explicit; and a user
forgetting to opt a weak model in — accepted as the honest cost of not guessing
(a wrong guess, e.g. terse-on-weak, is a silent correctness failure; a missing
opt-in is visible and one setting away).

**Alternatives rejected.** *Per-model monolith selection (opencode's model)* —
proven and simple to reason about per model, but duplicates shared rules across
files and abandons MECE composition; rejected as the primary mechanism, retained
only as the narrow escape hatch for cross-cutting voice. *Weak = drop sections* —
the intuitive read, but contradicted by the evidence (weak models need *more*
explicit/repeated/exemplified text, not less); modeling strength as section count
would build the wrong thing. *Keep a single register and just trim for weak
models* — loses the strong-model density that works well today and still can't add
the examples/repetition weak models need. *Full per-vendor variant matrix
(`persona.gpt.md`, `persona.gemini.md`, …)* — maximal fidelity, but the
maintenance bomb that the shared/variant split exists to prevent. *Infer the
register from the model id (substring/family `provider()`-style mapper)* — the
original draft of this ADR; rejected after it misclassified DeepSeek V4 Pro (a
frontier model) as weak. A model id does not encode capability — families span
tiny→frontier — so the "mapper" guesses. It also contradicts `capabilities.py`,
which curates per-model with conservative unknown defaults rather than
family-matching. Replaced by an explicit, user-declared registry.

**Evidence.** Study basis: opencode's prompt set — per-family prompt files
selected by a family switch in `session/system.ts` — whose weaker-family prompts
run longer and more repetitive than the strong-family one (the register contrast
this ADR rests on). Implementation: the profile resolver and declared-mapping
registry (`prompt/profile.py`), variant-aware prompt loading (`prompt/prompt.py`),
profile threading through composition (`prompt/manager.py`), the `LLM_PROFILE`
knob (`config/mixins/llm_prompt.py`), and the first variant/examples files under
`prompt/markdown/`. Tests under `test/llm/prompt/`. Builds on ADR-0061
(config-positioned sections); the declared per-model registry mirrors the
`capabilities.py` model table. [DOCUMENTED] in code and this ADR.
