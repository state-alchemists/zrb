# ADR-0052 — Tool-output truncation backstop

**Status:** Accepted

**Context.** Only `Read`/`Bash` capped their own output; Grep, AnalyzeCode, web,
and MCP results were uncapped, so a huge result could flood the context window.

**Decision.** A global, deterministic cap (`CFG.LLM_MAX_TOOL_RESULT_CHARS`,
default 100k; `0` disables) applied in the tool wrappers, truncating the
model-facing `content` (directional — head-keep for file reads, tail-keep
for shell output) while leaving the structured `return_value` whole. A
typical result is under the cap and untouched. Truncation snaps to line
boundaries; shell output exceeding the cap is also dumped to a temp file
for later recovery via Grep/Read.

**Consequences.** Pathological outputs no longer degrade the model; programmatic
consumers still get full data. Cost: above the cap, the model sees a truncated
view and must narrow its query.

**Alternatives rejected.** Chunked streaming (runner surgery); per-tool caps
(every tool differs); no cap (latent context-flooding bug).

**Evidence.** `src/zrb/util/truncate.py` (`truncate_text`, `truncate_items`),
`src/zrb/llm/agent/truncate.py`, `agent/common.py`
(wrapper integration), `config/mixins/llm_limits.py`.

**Later revision (Simplify tools, v2.x).** The original head+tail-line
preservation algorithm (`truncate_output`) was replaced by a simpler
char-budget directional model: `truncate_text(text, max_chars, keep="head"|"tail")`
and `truncate_items(items, max_chars)`. Per-line length capping,
middle-elision, and `TruncationInfo` metadata were removed. Shell tools
gained temp-file dumping for full-output recovery.
