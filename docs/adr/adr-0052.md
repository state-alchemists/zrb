# ADR-0052 — Tool-output truncation backstop

**Status:** Accepted

**Context.** Only `Read`/`Bash` capped their own output; Grep, AnalyzeCode, web,
and MCP results were uncapped, so a huge result could flood the context window.

**Decision.** Two cooperating layers, both deterministic:

1. **Per-tool self-capping** (`CFG.LLM_MAX_OUTPUT_CHARS`, default 100k). The
   file/list/search/shell tools bound their *own* output *directionally* via
   `truncate_text` / `truncate_items`: `Read`, `List`, and `Grep`-style tools
   keep the head (`...[TRUNCATED]`); `Shell`/`Bash` keep the tail
   (`[TRUNCATED]...`, where errors and exit summaries land). Cuts snap to a line
   boundary, and `Shell`/`Bash` additionally dump the full stdout/stderr to a
   temp file whose path is surfaced for recovery via Grep/Read.
2. **Global backstop** (`CFG.LLM_MAX_TOOL_RESULT_CHARS`, default 100k; `0`
   disables). Applied in the tool wrapper (`agent/common.py`) to *every* tool's
   model-facing `content` via `truncate_tool_content` — this catches the tools
   that don't self-cap (AnalyzeCode, web fetches, MCP toolsets). It keeps the
   head **and** tail with a middle marker describing how to re-fetch a narrower
   slice, and never touches the structured `return_value`, so programmatic
   consumers still get full data.

A typical result is under both caps and untouched.

**Consequences.** Pathological outputs no longer degrade the model; programmatic
consumers still get full data. Cost: above the cap, the model sees a truncated
view and must narrow its query.

**Alternatives rejected.** Chunked streaming (runner surgery); per-tool caps
(every tool differs); no cap (latent context-flooding bug).

**Evidence.** Per-tool caps: `src/zrb/util/truncate.py` (`truncate_text`,
`truncate_items`) wired into `llm/tool/file_read.py`, `shell.py`,
`file_list.py`, `file_search.py`. Global backstop:
`src/zrb/llm/agent/truncate.py` (`truncate_tool_content`) via `agent/common.py`.
Knobs `LLM_MAX_OUTPUT_CHARS` / `LLM_MAX_TOOL_RESULT_CHARS` in
`config/mixins/llm_limits.py`.

**Later revision (Simplify tools, v2.42.0).** The original per-tool head+tail
line-preservation algorithm (`truncate_output`, with per-line length capping,
middle-elision, and `TruncationInfo` metadata) was replaced by the simpler
char-budget *directional* model above (`truncate_text` / `truncate_items`), and
the `LLM_FILE_READ_LINES` knob that drove its line counts was removed in favour
of `LLM_MAX_OUTPUT_CHARS`. `Shell`/`Bash` gained temp-file dumping for
full-output recovery. Note this directional change is per-tool only — the global
backstop (`truncate_tool_content`) still keeps head **and** tail with a middle
marker.
