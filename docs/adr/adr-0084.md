# ADR-0084 — Named style themes: one `ZRB_THEME` preset supplies defaults for every style knob

**Status:** Accepted (relates to ADR-0021 mixins, ADR-0023 precedence)

**Context.** zrb's visual styling was spread across three groups of config:
the LLM chat TUI (`LLM_UI_STYLE_*`, prompt_toolkit style strings), the CLI
semantic colors (`CLI_COLOR_*`/`CLI_STYLE_*`, Rich color names), and the
markdown renderer. Two problems:

1. **No coordinated defaults.** Each style knob carried its own `DEFAULT_*`
   constant. Restyling the whole tool (e.g. for a light terminal) meant setting
   ~40 individual `ZRB_*` env vars by hand — there was no way to say "use a
   light palette" once.
2. **Markdown was not configurable at all.** `util/cli/markdown.py` built a
   hardcoded `rich.Theme` (`markdown.link`, `markdown.h1`, `markdown.code`, …).
   Several other strings were hardcoded outside config too — the title-bar
   background (`ansipurple`), the input-prompt color, the choice-widget shades.
   These blocked any coherent alternate palette.

The aim: pick a theme by name and have it supply sensible defaults everywhere,
while keeping every individual knob independently overridable — and change
nothing for existing installs.

**Decision.** Add a **theme axis** parallel to the existing per-knob env
resolution, reusing the `EnvField` precedence chain (ADR-0023: env var →
`default_factory` → attribute default) rather than inventing new machinery.

1. **A theme is a named palette.** `config/theme.py` holds
   `THEMES: dict[str, dict[str, str]]` mapping a theme name to `{knob_name:
   value}`. Ships `dark` and `light`. `dark` reproduces the historical
   hardcoded values **exactly**, so a default install is byte-for-byte
   unchanged. The module is pure data + helpers and imports no config, to keep
   `zrb.config` cycle-free.

2. **One selector knob.** `ZRB_THEME` (`CFG.THEME`, default `dark`) picks the
   palette. It spans all three style groups — a single knob themes the whole
   tool, even though the palette dict mixes prompt_toolkit and Rich
   vocabularies (each key maps to exactly one knob with a known consumer, so the
   mixing is safe).

3. **Every style knob resolves its default from the active theme.** Each field
   is wired `EnvField(default_factory=theme_default("KNOB_NAME"))`, where
   `theme_default` returns `lambda host: theme_value(host.THEME, key)`. Because
   `EnvField` reads the env var first and only calls the factory when unset, an
   explicit `ZRB_LLM_UI_STYLE_*`/`ZRB_CLI_*` env **always wins over the theme**.
   The per-knob `DEFAULT_*` constants are removed — the theme dict is now the
   single source of truth for defaults.

4. **Markdown and the stragglers become real knobs.** New
   `LLM_UI_STYLE_MARKDOWN_{LINK,LINK_URL,H1,CODE}` feed `render_markdown`'s theme
   when no explicit `Theme` is passed. New `LLM_UI_STYLE_TITLE_BAR_BG` and
   `LLM_UI_STYLE_PROMPT` lift the previously hardcoded title-bar background and
   input-prompt color; the jargon/hint/choice shades were repointed at existing
   `FAINT`/`TEXT` knobs. This is what makes `light` actually coherent.

5. **User-extensible.** `register_theme(name, values)` (mirroring
   `register_model_profile`, ADR-0083) lets `zrb_init.py` add or override
   themes. Partial themes are fine — a missing key resolves to `""`. An unknown
   `ZRB_THEME` logs a warning and falls back to `dark`.

**Consequences.** Default behavior is unchanged (`dark` == the old constants).
Users get whole-tool restyling from one env var and can still override any
single color. Adding a knob to the theme system means: add the field with
`default_factory=theme_default(...)` and add its value to **both** built-in
themes. New surface: `config/theme.py`, the `ThemeMixin`, and the
`register_theme` hook. Risks: *palette drift* — a knob added to `dark` but not
`light` would blank that style under `light`; mitigated by a parametrized test
asserting both themes cover the same key set. *Vocabulary confusion* — a Rich
color placed in a prompt_toolkit knob (or vice-versa) renders wrong; accepted,
since each key has a single fixed consumer and is exercised by the UI tests.

**Alternatives rejected.** *Keep per-knob defaults, ship a "light preset" as a
documented env-var dump* — no code change, but users must paste ~40 vars and
there is no single toggle; abandons the "works by default, one knob to
restyle" goal. *A dedicated Theme object threaded through the UI instead of
`default_factory`* — cleaner in isolation but bypasses the `EnvField`
precedence that already gives per-knob overrides for free, and would need every
consumer rewired; the `default_factory` approach reuses ADR-0023 wholesale.
*Separate theme knobs per group (`ZRB_LLM_THEME`, `ZRB_CLI_THEME`)* — more
granular but defeats the "theme the whole tool at once" intent; a single knob
with `register_theme` for special cases is simpler. *A dict-valued `EnvField`
parsing `ZRB_THEME_OVERRIDES={...}`* — no dict cast exists at the env layer and
delimited-scalar parsing of style maps is brittle; per-knob env overrides
already cover ad-hoc tweaks.

**Evidence.** Implementation: the palette registry and helpers
(`config/theme.py`), the selector (`config/mixins/theme.py`), themed fields
(`config/mixins/llm_ui_styles.py`, `config/mixins/cli_style.py`), the
CFG-driven markdown theme (`util/cli/markdown.py`), and the lifted TUI strings
(`llm/app/layout.py`, `llm/app/style.py`, `llm/ui/default/output_mixin.py`).
Tests: `test/config/test_config_theme.py`, `test/util/cli/test_markdown.py`.
Builds on ADR-0021 (mixin composition) and ADR-0023 (resolution precedence);
the user-declared registry mirrors ADR-0083's `register_model_profile`.
