# ADR-0076 — Uniform `add_hook_factory` for task-level hook registration

**Status:** Accepted

**Context.** `LLMChatTask` and `LLMTask` each compose a hook manager, but their
lifecycles differ. `LLMChatTask` builds a **fresh** `HookManager` on every
execution and replays the registered hook factories onto it. `LLMTask` holds a
**persistent** manager that defaults to the process-wide shared `hook_manager`
singleton and is passed straight to `run_agent`. Only `LLMChatTask` had a
task-level registration API (`add_hook_factory`); registering hooks on an
`LLMTask` after construction had no first-class path.

**Decision.** Both classes register hooks through one uniform verb:
`add_hook_factory(*factory)` / `append_hook_factory(*factory)`, where each
`factory: Callable[[HookManager], None]` registers hooks on the manager it is
handed (typically `mgr.register(hook, events=[…])`, which takes effect
immediately — the run path reads the registered hooks directly and does not rely
on a `scan()`).

- **`LLMChatTask`** keeps its existing factory list, replayed onto the fresh
  per-execution manager. Unchanged.
- **`LLMTask`** gains `add_hook_factory`/`append_hook_factory`, applied
  immediately to its manager. To keep task hooks **isolated by default**, the
  first registration swaps the shared global default for a fresh per-task
  `HookManager` (`_ensure_task_local_hook_manager`). A manager supplied
  explicitly via the `hook_manager=` constructor argument is never swapped, so
  opting into the global (or any specific) manager stays possible.

`LLMTask` also gains a `history_manager` get/set property, symmetric with the
one `LLMChatTask` already had.

**Consequences.** One mental model — "register a hook factory" — works on both
task types, even though the implementations differ underneath (replay-per-exec
vs apply-once to a task-local manager); the contract is identical. `LLMTask`
hooks are leak-free by default: registering no longer mutates the shared global
singleton, matching `LLMChatTask`'s per-execution isolation. The price of that
isolation is that an `LLMTask` registering via `add_hook_factory` no longer
participates in the global/filesystem hook set unless it is explicitly
constructed with the global manager — called out in the method docstring.

**Alternatives rejected.** `LLMTask.add_hook_factory` applied to the global
default (symmetric API, asymmetric safety — silently mutates state shared by
every task; the lazy task-local swap avoids this). A task-level
`add_hook_factory` that defers to `HookManager.scan()` (the run path never scans,
so registrations would never fire).

**Evidence.** **[DOCUMENTED]** `src/zrb/llm/task/llm_task.py`
(`add_hook_factory`/`append_hook_factory`, `_ensure_task_local_hook_manager`,
`history_manager` property; `run_agent(hook_manager=self._hook_manager)`);
`src/zrb/llm/task/chat/builder_mixin.py` (`add_hook_factory`/
`append_hook_factory`, unchanged); `src/zrb/llm/hook/manager/manager.py`
(`register` immediate; `add_hook_factory` defers to `scan()`, which the run path
does not call). `test/llm/task/test_llm_task.py`
(`test_add_hook_factory_isolates_and_reaches_runner` — behavioral isolation +
reaches-runner). Related: ADR-0061 (prompt-section programmability), ADR-0074
(hook capability parity).
