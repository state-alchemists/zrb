# ADR-0079 — Fold the skill catalogue into the `mandate` section (drop `claude_skills`)

**Status:** Accepted (refines ADR-0035, ADR-0069)

**Context.** The skill catalogue lived in its own MECE prompt section,
`claude_skills` (ADR-0035), which listed every model-invocable skill plus the
full content of any pre-activated skills. But the *policy* that governs
activation — when to activate, authority, tie-breaks — lived in the `mandate`
section's "Skill Activation" rules, and `mandate` *also* carried a hard-coded
table of the four `core-*` deliverable skills. So the same information was
described in two sections: the activation policy and core-skill table in
`mandate`, the catalogue in `claude_skills`. That splits one behavior ("which
skills exist and when to activate them") across two sections — the opposite of
the MECE principle ADR-0035 exists to enforce — and let the hand-maintained core
table drift from the actual `core_skills/` directory.

**Decision.** Remove the `claude_skills` section and render the catalogue inside
the mandate's Skill Activation rules via placeholder substitution. A new
`build_skill_replacements(skill_manager, active_skills)` (`prompt/claude.py`)
computes three `{PLACEHOLDER}` values, passed as `get_prompt` extra-replacements
when the `mandate` file section is composed:

- `{CORE_SKILLS}` — the always-on methodology baseline (built-ins under
  `llm_plugin/core_skills/`, detected by path), listed from their live
  descriptions. This replaces the hard-coded core-skill table.
- `{AVAILABLE_SKILLS}` — every other model-invocable skill (user/project/plugin).
- `{ACTIVE_SKILLS}` — full content of any pre-activated skills (the `active_skills`
  task param), empty when none; these are excluded from the two lists above.

The core/other split surfaces the ADR-0069 distinction (load-bearing core vs
optional extras) directly in the prompt. `claude_skills` is dropped from
`DEFAULT_LLM_INCLUDE_SECTIONS` and from the built-in agents' `inherit_sections`;
a config that still lists it is silently skipped (no warning) so pinned
`ZRB_LLM_INCLUDE_SECTIONS` values don't break. The
`create_claude_skills_prompt` middleware is removed in favour of
`build_skill_replacements`.

**Consequences.** The skill catalogue and the rules that govern it now live in
one section, restoring MECE; the core list is generated from `core_skills/`, so
it can't drift. One fewer section to order. The mandate stays byte-stable across
turns (the skill set is scan-cached and `active_skills` is a static task param),
so this does not regress prompt caching (ADR-0065). Cost: `mandate` is no longer
a purely static `.md` — it depends on `build_skill_replacements` at compose time;
and `create_claude_skills_prompt` is a removed public symbol (breaking for any
external caller importing it from `zrb.llm.prompt`).

**Alternatives rejected.** *Keep `claude_skills` as a separate section* — the
status quo, but it duplicates the mandate's activation rules and core table
across two sections (the MECE violation above). *Register the catalogue as a
config-positioned provider* (ADR-0061) — heavier machinery for content that is a
natural, always-on part of the mandate's structure; the placeholder path is
simpler and keeps the catalogue adjacent to its rules. *Auto-list all `core-*`
skills including `core-journaling` only in `journal_mandate`* — rejected;
`core-journaling` is listed uniformly in `{CORE_SKILLS}` (its activation trigger
belongs with the others), while `journal_mandate` still owns the protocol detail.

**Evidence.** `src/zrb/llm/prompt/claude.py` (`build_skill_replacements`,
`_is_core_skill`), `src/zrb/llm/prompt/manager.py` (extra-replacements wiring +
`claude_skills` skip), `src/zrb/llm/prompt/markdown/mandate.md` (placeholders),
`config/mixins/llm_prompt.py` (default order), `llm_plugin/agents/*.agent.md`
(`inherit_sections`), `AGENTS.md` (LLM Prompt System).
