# ADR-0082 — Journal index lives in `<live-context>`, not the cached system prompt

**Status:** Accepted

**Context.** The `journal_mandate` system-prompt section embedded a snapshot of
the journal index (`index.md`, truncated to 1000 chars) via a
`{JOURNAL_INDEX_CONTENT}` placeholder. The placeholder was filled by
`_get_prompt_replacements_cached`, whose `lru_cache` key included the index
file's `mtime` precisely so the snapshot refreshed when the journal changed.

That refresh was the problem. The system prompt is the cacheable prefix — every
other part of zrb's prompt assembly works to keep it byte-stable so provider
prompt caching survives across turns (this is the whole reason `<live-context>`
exists: volatile per-turn state is injected into the user turn, not the system
prompt — see the caching note in `live_context.py`). But the journal index is
*mutable mid-session*: the agent journals as part of its normal loop, the index
file's `mtime` changes, the cache key changes, and the next turn's system prompt
differs by those ~1000 chars. One journal write invalidated the entire cached
prefix for the rest of the session — the design defeated its own caching for the
sake of a content block that does not even need to be in the cached region.

A first fix attempt — "inject the index into the *first* turn's live-context
only" — was wrong: summarization (`split_history`) keeps only the last
`summary_window` messages verbatim and folds everything older into one summary
message (`/compress` collapses *everything*). The first turn is the first thing
dropped, so the index vanished after the first compaction and never returned.

A second attempt — "inject whenever a marker is absent from history" — worked but
was over-engineered: it reconstructed *when the index went missing* by scanning
history for a sentinel, when in fact the index can only go missing at two
**observable events**. Reconstructing an event you can observe directly is the
wrong shape.

**Decision.** Remove the index content (and the now-unused
`{CFG_LLM_JOURNAL_*_STATUS}` placeholders and `mtime` cache key) from the
`journal_mandate` markdown and from `_get_prompt_replacements_cached`. The
section now carries only a stable pointer line naming the index file. The
snapshot (`render_journal_index`, `live_context.py`) is then injected **at the
two — and only two — moments the index can otherwise be absent**:

1. **First turn** (a fresh session) — `llm_task._exec_action_inner` passes
   `inject_journal_index = not message_history` to `render_live_context`, which
   appends the snapshot to the turn's `<live-context>`. Detection is just
   "history is empty" — deterministic, no scanning.
2. **Each summarization** — `summarize_history` bakes the snapshot into the
   summary text right before building the summary message, in the branch that
   only runs when compaction actually happens. `summarize_history` is the single
   choke point for both `/compress` (`force=True`) and auto-summarization (the
   history processor), so one hook covers both.

Between those events the index simply persists in history untouched, so no
per-turn work and no presence check are needed.

Both injection points are **coupled to the `journal_mandate` section**: the
index is content that section owns, so it is emitted only when the section is
active. The first-turn snapshot is gated on the section's presence, and the
summarizer takes the same gate as a flag from its caller. A prompt that drops
`journal_mandate` therefore drops the index everywhere — first turn and
post-compaction alike — instead of silently re-seeding it.

The mechanism also reaches **sub-agents**. A sub-agent is single-turn (one run,
empty history), so it is always "the first turn": when it inherits
`journal_mandate`, the snapshot is folded into its composed system prompt. If it
also inherits `system_context` the index rides along with the per-turn state;
otherwise only the index is folded in, so a `journal_mandate`-only sub-agent
does not pick up the per-turn state that `system_context` owns. Its
auto-summarizer carries the same coupling.

**Consequences.** The cached system-prefix is now byte-stable across journal
writes — caching survives a journaling session, which is the common case. The
index lives as a single in-history copy (≤1000 chars), not in the prefix and not
repeated every turn. There is **no detection logic** (no marker, no history
scan, no cross-turn flag) and **no one-turn blind spot**: because the history
processor runs *before* the model request, the summary that re-seeds the index is
present in the very same compacted request, not the next one. Cost: a small,
deliberate coupling — `summarize_history` imports `render_journal_index`, i.e.
the summarizer now knows the journal index is context worth preserving. This is a
narrow, documented dependency on a pure read-and-format function.

**Alternatives rejected.** *Keep it in the cached system prompt* — the status
quo; simplest, but invalidates the cacheable prefix on every journal write, the
exact cost the live-context mechanism was built to avoid. *Inject on the first
turn only* — loses the index after the first summarization (turn 1 is the first
message dropped). *Inject on every turn* — robust, but accumulates up to
`summary_window` stale copies in history and roughly doubles live-context size
every turn; the index is the largest live-context component, so the accumulation
is the one worth avoiding. *Inject once-per-window by scanning history for a
sentinel tag* — correct and thrifty, but reconstructs an observable event
(summarization) via after-the-fact detection, adds a per-turn history walk, and
has a one-turn blind spot (the scan runs at turn start, before that turn's
summarization); hooking the event directly is simpler and gap-free. *Track
re-seeding with an in-memory flag* — does not survive a conversation resumed in a
fresh process. *A persisted sidecar flag* — survives resume, but adds a new state
store for what injecting-at-the-event records in history for free.

**Evidence.** The snapshot read and its injection seam live in
`prompt/live_context.py`; the cached system-prompt section (`prompt/prompt.py`,
`prompt/markdown/journal_mandate.md`) no longer embeds the index. The two
injection points and their coupling to the active `journal_mandate` section are
in the prompt manager (live-context), the history summarizer (summarization),
and the LLM task (first turn / `/compress`); sub-agent inheritance is in the
sub-agent manager. Tests under `test/llm/prompt/`, `test/llm/summarizer/`, and
`test/llm/agent/subagent/`. [DOCUMENTED] in code comments and this ADR.
