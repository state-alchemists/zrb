# ADR-0077 тАФ Configurable semantic CLI color layer (`CLIStyleMixin`)

**Status:** Accepted

**Context.** `src/zrb/util/cli/style.py` provided physical-color helpers (`stylize_yellow`, `stylize_faint`, `stylize_cyan`, тАж) and a thin semantic layer (`stylize_warning`, `stylize_error`, `stylize_log`) that were hardcoded aliases of the physical functions. Call sites throughout the codebase (тЙИ90 across тЙИ21 files) called physical helpers directly тАФ meaning the meaning of a colour was scattered across every file rather than declared once. Users with non-standard terminal palettes (e.g. terminals where yellow is already reserved, accessibility needs, dark/light theme contrast) had no way to remap any of them.

The project already has a precedent for env-var-driven style config: `LLMUIStylesMixin` controls the prompt-toolkit colours for the TUI (hex/ANSI strings). The same pattern applies cleanly to ANSI escape-code colours for plain terminal output.

**Decision.** Split semantic colours into two layers:

1. **Physical layer** тАФ `stylize_yellow`, `stylize_red`, `stylize_green`, `stylize_blue`, `stylize_cyan`, `stylize_magenta`, `stylize_bold_*` тАФ unchanged. These are intentionally literal (e.g. git-diff convention: green=added, red=removed, yellow=modified). Call sites that need a specific colour by convention should keep using them.

2. **Semantic layer** тАФ a fixed set of meaning-named helpers, each backed by a `CLIStyleMixin` field. Defaults reproduce the previous hardcoded behaviour; users may override via `ZRB_CLI_*` env vars.

Semantic slots defined:

| Semantic name          | Default colour | Default style | Env var (prefix `ZRB_`)          |
|------------------------|---------------|---------------|----------------------------------|
| `stylize_muted`        | _(none)_      | faint         | `CLI_COLOR_MUTED` / `CLI_STYLE_MUTED`     |
| `stylize_warning`      | yellow        | bold          | `CLI_COLOR_WARNING` / `CLI_STYLE_WARNING` |
| `stylize_error`        | red           | bold          | `CLI_COLOR_ERROR` / `CLI_STYLE_ERROR`     |
| `stylize_success`      | green         | _(none)_      | `CLI_COLOR_SUCCESS` / `CLI_STYLE_SUCCESS` |
| `stylize_highlight`    | yellow        | bold          | `CLI_COLOR_HIGHLIGHT` / `CLI_STYLE_HIGHLIGHT` |
| `stylize_info`         | cyan          | _(none)_      | `CLI_COLOR_INFO` / `CLI_STYLE_INFO`       |
| `stylize_todo_project` | yellow        | _(none)_      | `CLI_COLOR_TODO_PROJECT`         |
| `stylize_todo_context` | cyan          | _(none)_      | `CLI_COLOR_TODO_CONTEXT`         |
| `stylize_todo_keyval`  | magenta       | _(none)_      | `CLI_COLOR_TODO_KEYVAL`          |

`stylize_faint` and `stylize_log` are kept as backward-compat aliases for `stylize_muted`.

All call sites in the codebase were migrated from physical helpers to their semantic equivalents. The git-diff colours in `builtin/git.py` (green/yellow/red for added/modified/removed) were intentionally left as physical calls.

The semantic helpers use lazy CFG imports (`# lazy: defer CFG load`) to avoid adding config machinery to the startup cost of every early `style.py` import.

**Consequences.**
- *Good.* Colour semantics (warning = yellow) declared once; every consumer inherits overrides automatically.
- *Good.* Consistent: follows the `LLMUIStylesMixin` pattern already in the codebase; config naming follows ADR-0073 rules (verb-first `CLI_COLOR_*` / `CLI_STYLE_*` for standalone on/off behaviours).
- *Good.* Zero breakage: physical helpers are unchanged; `stylize_faint`/`stylize_log` remain as aliases.
- *Cost.* Each semantic call pays one dict-lookup through `_resolve_color`/`_resolve_style` (negligible for CLI output).
- *Cost.* Lazy CFG import means the first call per semantic function triggers a `sys.modules` lookup (also negligible after warm-up).

**Alternatives rejected.**
1. **Hardcode the semantic layer (status quo)** тАФ users cannot remap colours and call sites proliferate with physical names. Rejected because the physical-vs-semantic split is already half-done in the existing `stylize_warning`/`stylize_error` functions.
2. **Use a module-level registry dict, populated at startup** тАФ avoids per-call CFG reads but requires explicit initialisation order and a public mutate API. More complex with no meaningful performance benefit for CLI output paths.
3. **Expose full ANSI integer codes as env vars** тАФ technically precise but not user-friendly (`ZRB_CLI_COLOR_WARNING=33`). Rejected in favour of colour-name strings mapped internally via `_COLOR_NAME_MAP`.

**Evidence.** **[DOCUMENTED]** `src/zrb/config/mixins/cli_style.py` (`CLIStyleMixin`); `src/zrb/util/cli/style.py` (`_COLOR_NAME_MAP`, `_STYLE_NAME_MAP`, semantic helpers); `src/zrb/config/config.py` (mixin composition). **[INFERRED]** git-diff physical-colour preservation visible in `src/zrb/builtin/git.py`.
