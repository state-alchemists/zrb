# ADR-0078 — First-class `permissions` parameter on `LLMChatTask`

**Status:** Accepted

**Context.** A permission policy reaches the agent through the
`current_permission_policy` ContextVar, which `run_agent` reads (`runner.py`)
and every tool call consults via `get_effective_policy()`
(`permission/state.py`). `LLMTask` already accepted a `permissions=` constructor
argument and forwarded the resolved policy to `run_agent`. `LLMChatTask` did
**not**: it had no `permissions=` argument and `_create_llm_task_core` did not
forward one to the inner `LLMTask`. The only programmatic way to give an
`LLMChatTask` a policy (short of the `ZRB_LLM_PERMISSIONS` env var) was to call
`set_current_permission_policy(...)` from inside a hook factory registered via
`add_hook_factory`:

```python
def apply_policy(manager):           # ignores `manager`
    set_current_permission_policy(my_policy)
    return manager
safe_chat.add_hook_factory(apply_policy)
```

This worked, but misused the API. `add_hook_factory` is for registering hook
**event handlers** (`mgr.register(hook, events=[...])`, see ADR-0076); here the
factory ignored its `HookManager` entirely and exploited only the fact that
factories run synchronously when the per-execution manager is built — before the
agent run — to set the ContextVar at the right moment. It read like a
`SessionStart` hook but was neither a hook nor tied to the `SESSION_START`
event, which repeatedly caused confusion (and the example/docs described it with
exactly that misleading framing). The two examples that needed it
(`examples/plan-mode`, `examples/permission-policy`) both carried this boilerplate.

**Decision.** Give `LLMChatTask` a first-class `permissions` parameter,
symmetric with `LLMTask`:

- `LLMChatTask.__init__` accepts `permissions=` and stores `self._permissions`.
- `_create_llm_task_core` forwards `permissions=self._permissions` to the inner
  `LLMTask`, which resolves it (`resolve_policy`) and passes it to `run_agent`
  as the explicit `permission_policy` argument.
- Both classes expose `permissions` as a read/write **property** (the
  setter pairs them with the existing `tool_confirmation` / `approval_channel` /
  `history_manager` properties; `LLMTask` previously had the constructor arg but
  no property).

Resulting precedence — explicit `permissions=` argument **>** `ZRB_LLM_PERMISSIONS`
env **>** ambient ContextVar set by a parent run (sub-agent inheritance). Plan
Mode's read-only preset still overrides all of them while active, because
`get_effective_policy` checks `agent_mode == PLAN` first.

The sibling `sandbox` knob (`SandboxPolicy`, `coerce_sandbox`) got the same
treatment for consistency: `LLMTask` already accepted `sandbox=` but exposed no
property, and `LLMChatTask` had neither. Both now accept `sandbox=` and expose a
`sandbox` property, and `LLMChatTask` forwards it to the inner `LLMTask` →
`run_agent(sandbox_policy=...)`.

The accepted input shapes are now named types — `PermissionPolicyInput`
(`PermissionPolicy | str | Sequence[Rule | dict] | None`) and `SandboxInput`
(`SandboxPolicy | bool | None`), exported from their packages — replacing the
prior `Any` on both the task arguments/properties and the `resolve_policy` /
`coerce_sandbox` coercion functions. The policy classes stay
`@dataclass(frozen=True)` with tuple fields: immutability is load-bearing
because `PLAN_MODE_POLICY` is a shared module-level singleton propagated through
a ContextVar and snapshotted per tool call; the read/write `permissions` /
`sandbox` properties give the "change it later" affordance by reassigning a new
frozen instance, never by mutating one.

The examples (`examples/plan-mode`, `examples/permission-policy`) configure the
built-in `llm_chat` task via `llm_chat.permissions = …` — the task `zrb llm chat`
actually runs — rather than constructing a separate `LLMChatTask` the documented
command would never invoke (a latent mismatch in the originals).

**Consequences.** The common case — "this chat task runs under this policy" — is
a one-liner (`LLMChatTask(permissions=my_policy)`), with no hook-factory
boilerplate and no need to know about the ContextVar. The `apply_policy` factory
was removed from both examples. The ContextVar remains the underlying mechanism
and the documented escape hatch for **dynamic** policies chosen at runtime
(`set_current_permission_policy`), and stays the channel by which sub-agents
inherit a parent run's policy — the explicit argument simply takes precedence
when both are present. `add_hook_factory` is no longer (ab)used for permission
injection, restoring it to the single meaning ADR-0076 gave it.

**Alternatives rejected.** Keep the hook-factory pattern and document it well
(rejected — documenting a workaround is strictly worse than removing the need
for one; the "looks like a SessionStart hook" confusion is inherent to the
shape, not the wording). A dedicated `set_permissions()` method instead of a
property (rejected — inconsistent with the sibling property-based setters).

`_create_llm_task_core` forwards `permissions=self._permissions`);
`src/zrb/llm/task/llm_task.py` (`permissions`/`sandbox` property/setter; existing
`_permissions` → `resolve_policy` → `run_agent(permission_policy=...)`, `_sandbox`
