# ADR-0080 — Mode cycle binds Shift+Tab only, with a Termux-detected Tab fallback

**Status:** Accepted (refines ADR-0075)

**Context.** ADR-0075 introduced the Shift+Tab mode cycle (normal →
auto-accept-edits → plan). The original wiring bound *both* `tab` and `s-tab` to
the same `cycle_mode()` handler. The stated reason was Termux: on that Android
terminal, Tab and Shift+Tab both emit byte `0x09`, so Shift+Tab is
indistinguishable from Tab and never arrives as a distinct key. Binding plain
Tab was the way to keep mode cycling reachable there.

The cost was that on *every* terminal — including desktop terminals where the
two keys are distinct — plain Tab also cycled the mode. That made Tab unusable
for anything else and surprised users who expect Tab to be inert (or to drive
completion) and only Shift+Tab to change mode, matching Claude Code. One
platform's limitation was imposed on all platforms.

**Decision.** Bind `s-tab` to `cycle_mode()` unconditionally, and bind plain
`tab` to it **only when running under Termux**. Termux is detected by a new
`is_termux()` helper (`config/helper.py`) — true when `TERMUX_VERSION` is set or
`PREFIX` contains `com.termux` (either signal suffices, surviving a stripped
environment that drops one). The result is exposed as an auto-detecting,
env-overridable config knob `CFG.IS_TERMUX` (`EnvField` with
`default_factory=lambda c: on_off(is_termux())`, mirroring `SHELL`'s
auto-detection pattern). `setup_app_keybindings` adds the `tab` binding inside an
`if CFG.IS_TERMUX:` guard; both bindings keep the existing
`no_active_choice & ~has_completions` filter so a completion menu or choice
widget still owns Tab.

Because detection is an overridable knob rather than a hard-coded check, a user
on any terminal where Shift+Tab doesn't survive (not just Termux) can force
`ZRB_IS_TERMUX=true` to get the Tab fallback; conversely `ZRB_IS_TERMUX=false`
disables it even on a real Termux.

**Consequences.** Desktop terminals get the expected behavior — only Shift+Tab
cycles, Tab is free. Termux users keep a working mode cycle via Tab. The escape
hatch covers misdetection in both directions without a code change. Cost: one
more public config attribute (`CFG.IS_TERMUX`) and a platform branch in the
keybinding setup; the help text and status badge now advertise only "Shift+Tab",
which is accurate off Termux but understates the Tab fallback on Termux (deemed
acceptable — the badge is shared and Termux is the minority surface).

**Alternatives rejected.** *Keep binding both keys everywhere* — the status quo;
simplest, but imposes Termux's limitation on all terminals and makes Tab
unusable for completion/insertion. *Bind only Shift+Tab, no fallback* — clean
for desktop but silently removes mode cycling on Termux, a regression for those
users. *Detect Termux inline at the call site without a config knob* — works, but
gives no override for misdetection and buries a reusable platform fact in the UI
layer; promoting it to `CFG.IS_TERMUX` keeps detection with the other
environment-derived config and makes it testable and overridable.

**Evidence.** `src/zrb/config/helper.py` (`is_termux`), `config/mixins/foundation.py`
(`IS_TERMUX` field), `src/zrb/llm/ui/default/keybindings_mixin.py` (`s-tab`
unconditional + Termux-guarded `tab`), `src/zrb/llm/ui/base/commands_mixin.py`
(help text), `test/config/test_config.py` and
`test/llm/ui/default/test_keybindings_mixin.py` (coverage). [DOCUMENTED] in the
keybinding comments and this ADR.
