## 2.42.0 (June 24, 2026)

- **Breaking: removed the `LLM_FILE_READ_LINES` config knob** (`src/zrb/config/mixins/llm_content.py`): `LLM_FILE_READ_LINES` / `ZRB_LLM_FILE_READ_LINES` (and `DEFAULT_LLM_FILE_READ_LINES`, default `1000`) drove the old head/tail line-count truncation across the file tools. All file/list/search tools now bound output by `LLM_MAX_OUTPUT_CHARS` instead, so this setting no longer exists — remove it from any `ZRB_*` config.

- **Improvement: `truncate.py` rewritten around directional char-budget truncation** (`src/zrb/util/truncate.py`): The complex multi-step `truncate_output()` (head/middle/tail line elision with a `TruncationInfo` TypedDict and a half-dozen private helpers) is removed. It is replaced by `truncate_text(text, max_chars, keep="head"|"tail")` — a single char-cap that drops the overflow from one end and snaps the cut to a line boundary (`keep="head"` for file content, marking `...[TRUNCATED]`; `keep="tail"` for shell output where errors land at the bottom, marking `[TRUNCATED]...`) — plus `truncate_items(items, max_chars)` for head-keeping a list within a char budget (always keeps the first item). `truncate_chars()` is unchanged apart from its docstring.

- **Improvement: `Read` now takes an explicit line range instead of head/tail auto-truncation** (`src/zrb/llm/tool/file_read.py`): `read_file` replaces the `auto_truncate: bool` flag with `start_line` / `end_line` parameters (1-indexed, inclusive; `end_line=-1` reads through the last line, so the default still reads the whole file). A new `_validate_range` rejects inverted or out-of-bounds ranges with actionable `[SYSTEM SUGGESTION]` errors. Output over `LLM_MAX_OUTPUT_CHARS` is truncated from the end (head-keep), and the `[File: …]` header now reports the exact line span and whether the read was clipped — so a partial read is never mistaken for the full range. The global truncation backstop's recovery hint (`src/zrb/llm/agent/truncate.py`) was updated to point at `start_line`/`end_line` rather than the removed `offset`/`limit`.

- **Improvement: `Shell`/`Bash` truncate from the tail and dump full output to a temp file** (`src/zrb/llm/tool/shell.py`, `bash.py`): The `preserved_head_lines` / `preserved_tail_lines` parameters are removed; `max_chars` now defaults to `-1` (sentinel meaning "use `LLM_MAX_OUTPUT_CHARS`"). Command output exceeding the cap is now truncated from the TOP (keeping the tail, where errors and exit summaries land), and the complete stdout/stderr is written to a temp file (`zrb_shell_*.log`) whose path is reported via `[SYSTEM SUGGESTION]` so the elided head stays recoverable with Grep/Read.

- **Improvement: file listing and search bound output by char budget, not line counts** (`src/zrb/llm/tool/file_list.py`, `file_search.py`): `list_files`, `glob_files`, and `search_files` drop the `auto_truncate` parameter and the head+tail line-split logic. They now keep leading entries within `LLM_MAX_OUTPUT_CHARS` via `truncate_items`, and truncation notices read "first N of M". Search adds two explicit constants — `_MAX_MATCHES_PER_FILE = 100` (head-keep cap per file so one busy file can't dominate) and `_MAX_LINE_LENGTH = 1000` (per-snippet cap) — replacing the derived `LLM_FILE_READ_LINES // 4` figures.

- **Improvement: `analyze_file` drops its `auto_truncate` parameter** (`src/zrb/llm/tool/file_analyze.py`): `analyze_file` no longer accepts `auto_truncate`; it reads the file with the new range-based `read_file` defaults and clips to the analysis token budget via `truncate_text`.

- **Improvement: compact JSON schemas for `DelegateToAgent` and `RunZrbTask`** (`src/zrb/llm/tool/delegate.py`, `zrb_task.py`): `non_goals`/`tasks` (delegate) and `args` (zrb_task) keep mutable `= []` / `= {}` defaults (with `# noqa: B006` and an explanatory comment) rather than switching to `X | None = None`. pydantic-ai converts these to `default_factory` internally, so each call still gets a fresh container, while the tool JSON schema avoids the `anyOf: [type, null]` union that bloats every tool description sent to the LLM — continuing the 2.41.0 schema-reduction work.

- **Improvement: trimmed LLM tool docstrings to cut tool-description tokens** (`src/zrb/llm/tool/bash.py`, `shell.py`, `delegate.py`, `file_edit.py`, `file_write.py`, `plan.py`, `web.py`, `worktree.py`, `search/brave.py`, `search/searxng.py`, `search/serpapi.py`): Verbose `Args:`/`MANDATES:` blocks and prose were compressed to concise behavior summaries across the tool surface (e.g. `Bash` now defers to `Shell` for shared semantics; the three search backends collapse their bullet "MANDATES" into one line). Same behavior, fewer tokens spent describing tools on every request.

- **Improvement: skill catalogue folded into the `mandate` section** (`src/zrb/llm/prompt/markdown/mandate.md`, `src/zrb/llm/prompt/claude.py`, `manager.py`): The standalone `claude_skills` prompt section is removed. The skill catalogue is now rendered inside the mandate's Skill Activation section via `{CORE_SKILLS}`, `{AVAILABLE_SKILLS}`, and `{ACTIVE_SKILLS}` placeholders (`build_skill_replacements`). Core skills (the always-on baseline under `llm_plugin/core_skills/`) are listed separately from other model-invocable skills, and the hard-coded core-skill table in the mandate is replaced by the live list. `claude_skills` is dropped from `DEFAULT_LLM_INCLUDE_SECTIONS` and from the built-in agents' `inherit_sections`; a pinned config that still lists it is silently ignored rather than warning. `create_claude_skills_prompt` is removed from `zrb.llm.prompt`; use `build_skill_replacements` instead.
