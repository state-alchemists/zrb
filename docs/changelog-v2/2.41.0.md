## 2.41.0 (June 24, 2026)

- **Feature: `add_system_context()` and `add_project_context()` on `PromptManager`** (`src/zrb/llm/prompt/manager.py`): Two new public API methods for injecting dynamic content into specific built-in sections without editing `PromptManager` directly. `add_system_context(name, provider)` appends provider output after the built-in OS/CWD/tool/model block inside `system_context`; `add_project_context(name, provider)` appends after the built-in project documentation inside `project_context`. Both follow the same re-registration (overwrite by name) and silent-error contract as `add_live_context`. This completes the extension surface: plugins can now target `live_context`, `system_context`, or `project_context` for per-turn dynamic content, and any other position via `register_section`.

- **Fix: removed AGENTS.md stub-detection and auto-inlining from prompt composition** (`src/zrb/llm/prompt/claude.py`): The prompt system previously detected "stub" docs (files whose body is only `@path` references) and loaded them inline during composition, which double-counted content already present in `project_context`. The stub-detection regex, `_is_stub_doc`, `_load_file_content`, and `_load_file_content_cached` are removed. AGENTS.md and CLAUDE.md are no longer silently inlined at compose time.

- **Improvement: mandate — mandatory `Read` of project documentation files** (`src/zrb/llm/prompt/markdown/mandate.md`): Added a "Project Documentation" section requiring the agent to `Read` `AGENTS.md`, `CLAUDE.md`, and `README.md` in full on the first code-touching turn. A keyword search or grep does not satisfy this rule. Explicit paths in `project_context` take precedence; otherwise the agent probes the repo root directly.

- **Improvement: LLM tool parameter schema reduction** (`src/zrb/llm/tool/shell.py`, `bash.py`, `search/brave.py`, `search/serpapi.py`, `search/searxng.py`, `delegate.py`, `zrb_task.py`): Twelve optional parameters typed as `X | None = None` across the LLM-exposed tool functions have been converted to non-nullable defaults (`str = ""`, `int = 0`, `list = []`, `dict = {}`), treating the sentinel as equivalent to `None`. This eliminates `anyOf: [type, null]` unions from the tool JSON schemas — smaller schemas mean fewer tokens consumed per tool call. `safe_search: int | None = None` in `searxng.py` is left unchanged (0, 1, 2 are all valid values; no safe int sentinel exists).
