## 2.40.0 (June 21, 2026)

- **Improvement: first-class `permissions` and `sandbox` on `LLMChatTask`** (`src/zrb/llm/task/chat/task.py`, `src/zrb/llm/task/llm_task.py`, ADR-0078): `LLMChatTask` now accepts `permissions=` and `sandbox=` constructor arguments (symmetric with `LLMTask`) and forwards them to the inner `LLMTask` → `run_agent`. Both task types also expose `permissions` and `sandbox` as read/write properties. Previously the only programmatic way to give an `LLMChatTask` a `PermissionPolicy` was to call `set_current_permission_policy(...)` from a no-op hook factory — a pattern that misused `add_hook_factory` (which is for hook-event registration) and read misleadingly like a `SessionStart` hook. Precedence is explicit `permissions=` > `ZRB_LLM_PERMISSIONS` > ambient ContextVar (sub-agent inheritance); Plan Mode still overrides all. The `permissions`/`sandbox` arguments and `resolve_policy`/`coerce_sandbox` are now typed with exported `PermissionPolicyInput` and `SandboxInput` aliases instead of `Any`.

- **Fix: permission/plan-mode examples configure the built-in `llm_chat`** (`examples/plan-mode/zrb_init.py`, `examples/permission-policy/zrb_init.py`): both examples now set `llm_chat.permissions` on the built-in chat task that `zrb llm chat` actually runs, instead of constructing a separate `LLMChatTask` that the documented `zrb llm chat` command would never invoke. The [Permission Policy](../advanced-topics/permission-policy.md) doc gained a "Gating the built-in `zrb llm chat`" section; `set_current_permission_policy` remains the documented escape hatch for runtime-dynamic policies.
